# Camus Method Skill

The Camus Method establishes trust in AI-generated code through three phases: **Lexicon** (what the software manipulates), **Grammar** (how code must be written), and **Certification** (human signature).

## The Workflow

```
1. Define the Lexicon (terms and concepts)
2. Write code following Grammar rules
3. Review and verify compliance
4. Sign and certify (future: kiss sign)
```

---

## Phase 1: Lexicon

The Lexicon defines **what** the software manipulates. It is a finite, explicit set of *terms* shared between human and AI.

### Creating a Lexicon

When starting a project or new module:

1. **Identify core terms** — Ask: "What are the fundamental concepts this software deals with?"
2. **Define each term** — Write a clear definition in natural language
3. **Identify realizations** — Each term is realized by a component (class, struct, object)

### Lexicon Format

Create a `LEXICON.md` file at the project root:

```markdown
# Project Lexicon

## Term: [TermName]
- **Definition**: Clear description of the concept
- **Realized by**: ComponentName
- **Subterms**: 
  - SubtermName (property of ComponentName)
```

### Interaction with AI

When prompting the AI:
- Reference terms from the Lexicon explicitly
- Ask the AI to explain how it maps each term to components
- Verify that new code introduces only defined terms

## Phase 2: Grammar

Grammar rules ensure code is readable, auditable, and predictable. These are the 15 rules of the Camus Method:

### Core Rules (Must Follow)

1. **Term Definition** — Every term MUST be defined in the project Lexicon
2. **Component Realization** — Each component MUST realize exactly one term, explicitly declared
3. **Function Terms** — Every function MUST declare its input and output terms
4. **Function Claim** — Every function MUST declare what it does (its claim)
5. **Function Constraints** — Declare explicit, verifiable constraints on parameters
6. **Block Depth** — Functions MUST limit block depth to one level (no nested blocks)
7. **Line Length** — Lines MUST not exceed 120 characters (warn above 80)
8. **Function Length** — Functions MUST not exceed 50 lines (warn above 20)
9. **Parameter Passing** — All parameters MUST be passed by reference
10. **Variable Mutability** — Variables MUST be immutable by default

### Prohibition Rules

11. **No Public Primitives** — Never expose primitives as public variables or return values
12. **No Anonymous Functions** — Every function must have an explicit name
13. **No Inheritance** — Favor composition over inheritance
14. **No Interfaces** — Components communicate through explicit terms only

### Exception Rule

15. **Explicit Exceptions** — Deviations are allowed only if explicitly documented with the reason

### Verifying Grammar

For each function, verify:
- [ ] Does it declare its input/output terms?
- [ ] Does it state its claim?
- [ ] Is block depth limited to one level?
- [ ] Is it under 50 lines (ideally under 20)?
- [ ] Are variables immutable unless explicitly marked mutable?
- [ ] Are parameters passed by reference?
- [ ] Does it introduce only Lexicon-defined terms?

### When Writing Code

Guide the AI to follow these rules:
- Ask for function-by-function implementation
- Request explicit term declarations in comments
- Ask for claims to be stated at the start of each function
- Break long functions into smaller ones
- Prefer composition over inheritance patterns

### When Reviewing Code

1. Read each function and identify its terms
2. Verify the claim matches the implementation
3. Check block depth (no nesting beyond one level)
4. Count lines (flag if over 50)
5. Check for primitives exposed publicly
6. Verify immutability by default

---

## Phase 3: Certification

Certification is the human responsibility. When code passes Grammar review:

1. **Formatter** — Run formatter/linter for stable layout
2. **Grammar Check** — Verify all rules are respected
3. **Sign** — (Future: `kiss sign`) Each function signed by human with private key

### The Signature Promise

By signing a function, the human reviewer:
- Assumes responsibility for its correctness
- Guarantees the function performs its claimed action
- Commits to honoring the promise if challenged

## Quick Reference

### Lexicon Template

```markdown
# Lexicon

## [CoreTerm]
Definition: ...
Realized by: ...
```

### Function Header Template

```markdown
// Terms: inputTerm -> outputTerm
// Claim: [What this function does]
// Constraints: [Any constraints on inputs/outputs]
function [Name](param: InputTerm): OutputTerm {
  // implementation
}
```

### Review Checklist

- [ ] Lexicon exists and is complete
- [ ] Each function declares its terms
- [ ] Each function states its claim
- [ ] Block depth ≤ 1
- [ ] Function length ≤ 50 lines
- [ ] No public primitives
- [ ] Variables immutable by default
- [ ] No anonymous functions
- [ ] No inheritance
- [ ] Exceptions documented

## Interaction Pattern with AI

When working with AI using the Camus Method:

1. **Start with the Lexicon** — Define terms before writing code
2. **Request incremental delivery** — Ask for functions one at a time
3. **Demand explicit claims** — "Before writing this function, state what it claims to do"
4. **Verify on delivery** — Check each function against rules before proceeding
5. **Iterate and refine** — Adjust Lexicon as understanding deepens

The AI is a tool. The human is the author. Camus ensures the human masters what the AI produces.
